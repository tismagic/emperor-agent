# Scoring Rubric · 8 维评分尺

每个维度 0–10，**只用整数**。下面给每档的"硬证据"，避免主观浮动。

---

## 1. 项目结构

| 分  | 标准                                                                |
| --- | ------------------------------------------------------------------- |
| 10  | src-layout / monorepo 工作区 / 清晰分层 + namespace 包 + 多入口分离 |
| 8   | 顶层目录职责清晰，无同名脚本/包冲突，资源/代码分离                  |
| 6   | 主体清晰但缺 tests / Dockerfile / 入口与包同名                      |
| 4   | 文件散在根目录，业务/工具/配置混放                                  |
| 2   | 大量 .py 散在根目录，无包结构                                       |
| 0   | 单文件项目或 1 个 main.py 装下所有                                  |

**判定关键证据**：顶层 `ls` 输出 + 包含 `tests/` `src/` `pyproject.toml` 的对勾数。

---

## 2. 代码质量

5 个子项 × 2 分 = 10。

- **命名规范** (0–2)：snake_case / PascalCase / camelCase 一致；魔法字符串少
- **可读性** (0–2)：函数 ≤ 80 行；类 ≤ 300 行；复杂表达式有注释
- **可维护性** (0–2)：依赖明确、副作用集中、无隐式全局
- **解耦程度** (0–2)：抽象层次清晰；不同模块通过接口而非直接字段
- **可扩展性** (0–2)：新增同类对象（provider / handler / view）改动 ≤ 3 文件

**典型扣分**：单文件 535 行（-1）；构造函数 14 个参数（-1）；32 处 `print()`（-1）。

---

## 3. 架构设计

| 分  | 标准                                                         |
| --- | ------------------------------------------------------------ |
| 10  | 业内标杆，hexagonal / clean / DDD 真用对了；事件驱动；可插拔 |
| 9   | 分层清晰、依赖单向、抽象接口稳定，文档完整                   |
| 7   | 分层清晰但有 1–2 个上帝类；接口干净                          |
| 5   | 分层模糊，业务逻辑混进 controller / view                     |
| 3   | 没有分层，handler 直接调 DB / 文件 IO                        |
| 0   | 完全过程式 main.py                                           |

**判定**：画 import 拓扑图；找循环 import；查最大上帝类参数数。

---

## 4. 性能

| 分  | 评判要点                                             |
| --- | ---------------------------------------------------- |
| 10  | 有性能基准 + 监控指标 + 已优化热点                   |
| 8   | 无明显瓶颈，热点路径有缓存，DB 查询有索引            |
| 6   | 个人量级流畅，长期/高并发未验证                      |
| 4   | 已能预见 N+1 / 内存泄漏 / 无界队列                   |
| 2   | 多个明显瓶颈（同步阻塞 await / 全表扫描 / 重复计算） |
| 0   | 复杂度爆炸的算法或主线程阻塞                         |

**典型扣分**：`event_log: list = []` 无上限（-1）；`stats_by_*` 全量扫 jsonl（-1）。

---

## 5. 安全

| 分  | 评判要点                                                |
| --- | ------------------------------------------------------- |
| 10  | 通过 SAST + DAST + 渗透测试，明文密钥隔离，最小权限到位 |
| 8   | 鉴权 + 输入校验 + 密钥脱敏 + HTTPS 全到位               |
| 6   | 大部分到位但有 1 个中危（如 CORS 宽松）                 |
| 4   | 缺鉴权或密钥半泄漏，但有围栏                            |
| 2   | 存在已知高危：明文密钥回传、路径穿越、命令注入之一      |
| 0   | 多个高危叠加：RCE + 无鉴权 + 密钥裸出                   |

**经验**：A 类红旗（red-flags.md A1–A6）出现 1 条 ≤ 4；出现 2 条 ≤ 2。

---

## 6. 测试

最容易判：

| 测试覆盖率                             | 分  |
| -------------------------------------- | --- |
| 0 文件                                 | 0   |
| 1–10 个 case                           | 1   |
| 关键模块单测 + 部分集成                | 4   |
| line cov ≥ 60% + 关键路径 e2e          | 7   |
| line cov ≥ 80% + mutation testing      | 9   |
| + 性能测试 + 契约测试 + 完整 CI 跑全套 | 10  |

**recon.sh 的"测试文件总数"行直接出分**：0 → 0；< 5 → 1；< 20 → 3。

---

## 7. 工程化

按 recon.sh 第 1 节对勾数计：

| 对勾数 | 分   |
| ------ | ---- |
| 0–3    | 0–1  |
| 4–8    | 2–3  |
| 9–14   | 5–6  |
| 15–20  | 7–8  |
| 21+    | 9–10 |

必查：pyproject / lockfile / Dockerfile / CI 文件 / 至少一个 lint 配置 / pre-commit / .editorconfig / LICENSE / CONTRIBUTING / CHANGELOG / SECURITY。

---

## 8. 可维护性

是其他维度的"投影"，按以下加权：

```
maintain = 0.3*tests + 0.2*architecture + 0.2*engineering + 0.15*code_quality + 0.15*logging
```

加权后取整。换言之：测试 0、工程化 2 的项目，可维护性不可能 > 5。

---

## 综合评级

总分 = 8 维相加，最大 80。换 100 制 ÷ 0.8。

| 100 制 | 评级       | 说明                              |
| ------ | ---------- | --------------------------------- |
| 90+    | S 商业级   | 可上线、可外包、可融资讲架构      |
| 75–89  | A 良好     | 内部生产可用，新人 1 周内上手     |
| 55–74  | B 可优化   | 个人 / 小团队可用，商业化前需重构 |
| 35–54  | C 原型阶段 | 概念验证 ok，不要给陌生人用       |
| < 35   | D 高风险   | 重构成本可能高于重写              |

---

## 反对的打分姿势

- ❌ "整体 7 分" —— 不允许整体打分，必须分维度
- ❌ 给 6.5、7.8 这种 —— 整数即可
- ❌ 8 个维度全在 5–7 之间 —— 一定是没用心查
- ❌ 安全 9 但同时承认有未修的 RCE —— 自相矛盾
- ❌ 不带证据的 9/10 —— 比 6/10 更危险（误导用户决策）
